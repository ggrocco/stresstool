# gRPC Bidirectional Streaming Migration

**Date:** 2026-04-02
**Status:** Approved

## Summary

Migrate the controller-node communication layer from JSON-over-raw-TCP to gRPC bidirectional streaming. The primary goals are mTLS security, cleaner streaming semantics, a well-defined service contract via protobuf, and easier ecosystem extensibility. Protobuf encoding brings a modest performance improvement as a secondary benefit. Note: gRPC provides the transport and framing, but reconnection and health-checking require explicit configuration (keepalive parameters, retry policies) -- these are not automatic.

Additionally, introduce a `StopTests` message so the controller can cancel running tests mid-execution.

## Current State

- Controller opens a raw TCP listener; nodes dial TCP and keep the connection open.
- Messages use a JSON `Message{Type, Data}` envelope defined in `internal/protocol/protocol.go`.
- `json.Encoder` / `json.Decoder` on `net.Conn` handle framing.
- A `proto/api/v1/payload.proto` file already defines equivalent protobuf message types but is unused at runtime.
- `buf.gen.yaml` is already configured with `protocolbuffers/go` and `grpc/go` plugins, outputting to `internal/protocol/payloadpb/`.
- No TLS. No way to stop tests once started.

## Design

### Proto Service Definition

Add to `proto/api/v1/payload.proto`:

```protobuf
service StressTestService {
  rpc Session(stream NodeMessage) returns (stream ControllerMessage);
}

message NodeMessage {
  oneof payload {
    HelloMessage hello = 1;
    ReadyMessage ready = 2;
    ProgressMessage progress = 3;
    TestResultMessage test_result = 4;
    ErrorMessage error = 5;
  }
}

message ControllerMessage {
  oneof payload {
    TestSpecMessage test_spec = 1;
    StartTestsMessage start_tests = 2;
    CompleteMessage complete = 3;
    StopTestsMessage stop_tests = 4;
  }
}

message StopTestsMessage {
  string reason = 1;
}
```

The `oneof` replaces the type-dispatch pattern. All existing proto messages (`HelloMessage`, `ProgressMessage`, etc.) stay unchanged.

**Cleanup of legacy proto types:** The existing `Envelope` message and `MessageType` enum in `payload.proto` are removed. They were part of a never-used envelope design that the new `NodeMessage`/`ControllerMessage` oneofs fully replace. The JSON-based `protocol.go` types in Go are unaffected (they don't reference the proto file).

### Package Layout

```
proto/api/v1/payload.proto             -- UPDATED: service + oneof wrappers + StopTestsMessage
internal/
  protocol/
    protocol.go                        -- KEEP (standalone mode still uses JSON)
    payloadpb/                         -- GENERATED by buf: .pb.go + _grpc.pb.go
    convert.go                         -- NEW: Go types <-> protobuf converters
    convert_test.go                    -- NEW: round-trip conversion tests
  cli/
    controller.go                      -- REWRITE transport, keep state machine
    controller_grpc.go                 -- NEW: StressTestServiceServer implementation
    node.go                            -- REWRITE transport, keep test execution
    node_grpc.go                       -- NEW: gRPC client, Session stream management
    message.go                         -- REMOVE (replaced by protobuf)
    tls.go                             -- NEW: TLS config helpers
    web/                               -- UPDATE: add /api/stop endpoint + Stop button
```

**Key decisions:**

- **Runner and config stay protobuf-free.** A `convert.go` layer translates between `config.Config` / `runner.TestResult` and their protobuf equivalents. This keeps standalone mode (`stresstool run`) working without gRPC.
- **`message.go` is removed.** The `newProtocolMessage` / `decodeProtocolMessageData` helpers are replaced by generated protobuf types.
- **`protocol.go` survives** for standalone mode's JSON path. Can be removed later if standalone also migrates.

### Data Flow

```
NODE (gRPC client)                           CONTROLLER (gRPC server)
──────────────────                           ────────────────────────

Dial gRPC (with TLS) ──────────────────────► Accept connection
Open Session stream  ──────────────────────► Session() handler starts

HANDSHAKE:
  send NodeMessage{hello}  ───────────────►  register node in state manager
                                              node idles, waiting for start

... controller waits for user to type 'start' or trigger via web UI ...
... (nodes may connect/disconnect freely during this idle phase) ...

CONFIGURATION (triggered by 'start'):
  controller iterates registered nodes:
                            ◄───────────── send ControllerMessage{test_spec}
  validate config, apply
  node overrides
  send NodeMessage{ready}  ───────────────►  track node as ready (new state)

EXECUTION (controller waits for all ready, then sends start):
                            ◄───────────── send ControllerMessage{start_tests}
  start test runners

  send NodeMessage{progress} ─────────────► update state manager (every ~1s)
  send NodeMessage{progress} ─────────────►
  ...

  -- OR controller cancels --
                            ◄───────────── send ControllerMessage{stop_tests}
  cancel test context,
  drain workers,
  send partial results

  send NodeMessage{test_result} ──────────► store result (one per test,
                                             partial if stopped early)

TEARDOWN:
                            ◄───────────── send ControllerMessage{complete}
  close stream, disconnect
```

**Concurrency on the node side:** During execution, the node runs two goroutines per session:
1. A **send loop** that forwards progress and results via `stream.Send()`.
2. A **receive loop** that calls `stream.Recv()` to listen for `stop_tests` or `complete`. On receiving `stop_tests`, it cancels the runner's context.

This is a change from today where `executeTests()` blocks and the node doesn't listen for controller messages during execution.

**Concurrency on the controller side:** Each node's `Session()` handler runs in its own goroutine (standard gRPC). The controller must serialize `Send()` calls per stream -- only one goroutine may call `stream.Send()` at a time. Each session handler owns a **sendChan** (buffered channel); a dedicated goroutine drains it and calls `stream.Send()`. The controller state machine, the `/api/stop` HTTP handler, and the start logic all write to `sendChan` rather than calling `Send()` directly.

### Runner API Change for Cancellation

Today `runner.RunTest()` creates its own `context.WithDeadline(context.Background(), endTime)`. To support `stop_tests`, the signature changes to accept a parent context:

```go
func (r *Runner) RunTest(ctx context.Context, test *config.Test, progressChan chan<- ProgressUpdate) *TestResult
```

The runner derives its deadline from `ctx` using `context.WithDeadline(ctx, endTime)`. When the node receives `stop_tests`, it cancels the parent context, which propagates to all active `RunTest` calls (including parallel tests). Workers drain, metrics stop, and partial `TestResult` values are returned with whatever data was collected.

Standalone mode (`cli.go`) passes `context.Background()` as before -- no behavior change.

### Completion Semantics with Stop

When `stop_tests` is sent:
1. Nodes cancel their test contexts and send back partial `TestResultMessage` for each test.
2. The controller's `stateIsComplete` logic treats partial results the same as full results -- a result was received for that node+test pair.
3. After all nodes have reported (partial or full), the controller sends `complete` and prints the summary. Partial results are labeled in the summary output.

### Stream / Connection Error Handling

- **Node stream breaks (crash, network):** The controller's `Session()` handler receives an error from `Recv()`. It emits `evNodeDisconnected` to the state manager. If tests were running, that node is treated as failed (same as today's `evNodeError`). Partial results already received are preserved.
- **Controller stream breaks:** The node's `Recv()` returns an error. The node logs and exits. No automatic reconnection in v1 -- this matches current TCP behavior.
- **TLS handshake failure:** gRPC surfaces this as a connection error before the stream opens. Both sides log the error with the TLS failure reason.

### Web UI: Node Peer Address

Today `handleAPINodes` reads `nc.Conn.RemoteAddr()` to display node IP addresses. With gRPC there is no `net.Conn` on the session. Instead, the `Session()` handler extracts the peer address from gRPC's `peer.FromContext(ctx)` at handshake time and stores it alongside the node name in the state manager. The `/api/nodes` endpoint reads from this stored value.

### TLS / Security

Three modes selected by CLI flags:

| Mode | Controller flags | Node flags | Use case |
|------|-----------------|------------|----------|
| Insecure | `--insecure` | `--insecure` | Local dev |
| Server TLS | `--tls-cert`, `--tls-key` | `--tls-ca` | Encrypted, controller verified |
| Mutual TLS | `--tls-cert`, `--tls-key`, `--tls-ca` | `--tls-cert`, `--tls-key`, `--tls-ca` | Full mTLS for production |

Helper functions in `tls.go`:

```go
func ServerTLSConfig(certFile, keyFile, caFile string) (*tls.Config, error)
func ClientTLSConfig(certFile, keyFile, caFile string) (*tls.Config, error)
```

When `caFile` is provided on the server side, client certificates are required (mTLS). Users bring their own certificates.

### CLI Changes

`cmd/stresstool/main.go` gains shared TLS flags on both `controller` and `node` commands:

```
--tls-cert    Path to TLS certificate file
--tls-key     Path to TLS private key file
--tls-ca      Path to CA certificate for verifying the other side
--insecure    Skip TLS (plain gRPC, for local dev)
```

The `--listen` and `--controller` flags keep their current semantics. The transport changes from raw TCP to gRPC underneath.

### Web UI Changes

- New `POST /api/stop` endpoint on the controller, sends `StopTestsMessage` to all active node streams.
- "Stop" button added to the web interface alongside the existing "Start" button.

### Testing

**Unit tests (no network):**
- `convert_test.go`: round-trip `config.Config` → protobuf → `config.Config` and `runner.TestResult` → protobuf → `runner.TestResult`.
- State machine tests via channels remain unchanged.

**Integration test (in-process gRPC):**
- Use `bufconn` (in-memory listener) to avoid real ports.
- Full lifecycle: connect → hello → test_spec → ready → start → progress → result → complete.
- Stop test: send `stop_tests` mid-execution, verify partial results return.

**Existing tests unchanged:**
- `go test ./internal/runner` and `go test ./internal/config` have no gRPC dependency.
